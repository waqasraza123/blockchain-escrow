# Project State

## Product

`blockchain-escrow` is a programmable B2B milestone escrow platform for cross-border service and procurement deals. Drafting, evidence, review, disputes, operator workflows, and reporting live offchain. Custody, settlement, and critical money movement rules live onchain. The first lane targets Base and native USDC with a small contract surface.

## Current Architecture

- Monorepo with `pnpm` workspaces and `turbo`.
- Runtime apps: `apps/web` is now a server-rendered Release 10 customer workspace with hosted-session surfaces and read-only partner integration visibility, `apps/admin` is an internal operator console extended through Release 10 partner setup flows, `apps/api` is a NestJS control-plane API with org-user, operator, partner-key, and hosted-session surfaces, `apps/indexer` is the normalized chain-read service, and `apps/worker` now owns asynchronous draft activation, funding reconciliation, operator alerts, Release 9 finance-export artifact generation, and Release 10 partner webhook delivery side effects.
- Reusable packages: `packages/contracts`, `packages/contracts-sdk`, `packages/db`, `packages/shared`, `packages/security`.
- `packages/db` owns Prisma schema, migrations, generated client usage, and repository-backed persistence for Release 1, Release 2, Release 4 raw/projection entities, Release 8 operator/compliance/proposal entities, and generalized Release 9 approval/reporting/export entities.
- `packages/shared` owns cross-app types and validation schemas for Release 1 surfaces, Release 2 business contracts, Release 4 indexer contracts, Release 8 operator surfaces, generalized Release 9 approval/reporting/export surfaces, and shared primitives.
- `packages/security` owns auth/session/RBAC interfaces and helpers.
- `apps/api` now has Release 1 auth infrastructure wired with Prisma-backed persistence and nonce, verify, me, and logout endpoints.

## Non-Negotiable Rules

- Chain is the custody truth. Database is the operational truth.
- Drafts stay offchain until funding exists.
- The API never writes normalized chain projections directly.
- Only the indexer writes normalized chain projections and money-movement projections.
- Workers own side effects.
- Admin tooling must not bypass custody rules or gain arbitrary access to user escrow balances.
- Keep the contract surface small.
- Do not add fake escrow flows or placeholder money-movement logic.
- Preserve release sequencing from `docs/product/RELEASE_ROADMAP.md`.

## Important Decisions

- Prisma is the Release 1 persistence toolchain in `packages/db`.
- Default local Postgres runs on `127.0.0.1:5433` to avoid host `5432` collisions.
- Production chain is Base mainnet; development and staging use Base Sepolia.
- Initial token policy is native USDC only.
- `TokenAllowlist` intentionally stays minimal: later contracts should depend on `isAllowedToken(address)` rather than embedding token policy.
- The API owns offchain workflow state; the indexer owns normalized chain reads and projections.
- Future funded-escrow changes must come through new contracts and release work, not in-place mutation.
- Until the product grows a dedicated partial-refund deal state, `PARTIALLY_RELEASED` is the canonical non-terminal partial-settlement state for worker reconciliation, including refund-only partial outcomes.
- Until the contract surface grows a native dispute primitive, Release 7 dispute resolution remains an offchain arbitrator workflow that feeds the existing immutable milestone-settlement request path; chain custody execution still follows the same indexed settlement flow after the signed decision is accepted.
- Release 8 operator tooling is internal-only: operator accounts are seeded manually in the database, manual checkpoints/cases surface blocking state without yet hard-blocking customer funding or settlement APIs, and protocol proposal drafts stop at calldata generation rather than direct Safe or onchain submission.
- Release 9 approval policy resolution is deterministic and API-owned: exact action kind matches first, then cost-center specificity, then settlement-currency specificity, then newest update time.
- Release 9 keeps approval-control bootstrap exceptions intentionally narrow: org bootstrap/auth/invite-acceptance flows, signed counterparty/arbitrator submission flows, challenge-preparation endpoints, and approval-control mutations themselves are not approval-gated so the system cannot deadlock on its own control plane.
- Release 10 partner access is org-linked and additive rather than a separate workflow graph: partner-key writes reuse the existing org-owned workflow services through synthetic authenticated actors, approval gating still applies to partner mutations, operator setup remains the only write surface for partner accounts/links/keys/subscriptions, and customer org users only get read-only integration visibility in the web app.

## Deferred / Not Yet Implemented

- Later production-maturity and post-partner roadmap releases beyond Release 10 are not implemented.
- Worker-side deal lifecycle writes beyond indexed live activation plus funding-transaction reconciliation persistence are not implemented.
- Worker side effects such as notifications and operator-facing reconciliation tooling are not implemented.
- The Release 4 indexer foundation is implemented and the bounded local bootstrap/verify path now works in this workspace, but full historical replay to head still depends on the configured RPC capacity because public Base Sepolia endpoints can rate-limit large scans.

- Release 8 operator tooling is now implemented for internal use: the admin console, operator APIs, checkpoints/cases, and proposal drafting are production-structured, but operator accounts still require manual seeding, customer workflows are not yet hard-gated on checkpoints/cases, and protocol proposal drafts intentionally stop at calldata generation instead of Safe submission or execution.
- Release 9 approval/reporting/export surfaces are now implemented across the current customer workspace, but approval-control bootstrap exceptions remain by design for auth/bootstrap flows, signed counterparty/arbitrator submissions, challenge-preparation endpoints, and approval-control mutations themselves.
- Release 10 partner platform is now implemented, but partner setup remains intentionally operator-managed rather than self-serve, hosted session UX is intentionally narrow and workflow-scoped rather than a full embedded workspace, and webhook replay controls remain internal because the first delivery slice is at-least-once with worker-owned retries and inspection only.

- Release 4 projection correctness is covered by targeted indexer tests and the bounded local verification path now passes, but unbounded replay-to-head can still exceed public RPC limits without a higher-capacity provider.
